Tabs example

Tabs are an important element across Sky.com, in some places there are standard links to other pages styled as tabs, and some places where they dynamically swtich content on the same page.

NB: When we mean the tab key, we will use the KBD element styled like this: tab.

The accessibility concerns are with the dynamic switching of content.

Accessibility requirements

The basic requirements for tabs are:

  • Keyboard users can tab to the tabs, and select them.
  • There is visual indication of the selected tab.
  • There is a programmatic way for screen readers to understand what the tabs are, and which is selected.

Methods

There are two main methods for doing tabs:

1. Traditional method

The traditional method is where each tab is a heading-link at the top of each content block, with HTML such as: 

<div class="tabs">
   <h2>Tab heading 1</h2>
   <div class="tabbody">
     <p>Tab contents</p>
   </div>
   <h2>Tab heading 2</h2>

This works, although there is nothing to distinguish the links from standard links (that go to another page), it uses a dynamically assigned href to move the focus to the new content. This approach is exemplified by Ginader's accessible tabs (simple example).

2. WAI-ARIA method

The WAI-ARIA specification defines an interaction style for tabs, where the HTML structure lists the tab headings first, then shows the content:

<div id="tabs">
  <ul>
    <li><a href="#first">First Tab</a></li>
    <li><a href="#second">Second Tab</a></li>
    <li><a href="#third">Third Tab</a></li>
  </ul>
<div>
  <h2 id="first">First Tab's Main Heading</h2>
  <p>Lorem ipsum dolor sit amet...

The interaction style recommended is that the user uses the tab key to navigate to the set of tabs, and lands on the currently selected one. Rather than use the tab key, you need to use the arrow keys (left or right, and up or down) to select the different tabs.

In testing with people we've found few keyboard-users who would know to use the arrow keys for this purpose. (Screen readers are fine as the ARIA markup provides information to the screen reader.) There are two solutions to this issues:

  1. Add a visual highlight when the tabs receive focus, e.g. text pops-up saying "Use the arrow keys to select other tabs", perhaps using a cookie to save that the user has seen this hint.
  2. Enable the use of both tab and arrow keys. This works in simple examples of tabs, not nested menus.

For the moment we recommend the second method, which is the example below.

There are some key attributes that are added and manipulated by the script it is worth noting:

  • Add role=”tablist” to the <ul> to show it is the navigation for the tabs.
  • Add role=”tab” to each <a> in the list.
  • Add role=”tabpanel” to each <div> with content.
  • For the hidden <div>s add aria-hidden=”true”, and false for the one that's showing.
  • For the selected tab add aria-selected=”true”, and false for the others.
  • Ensure that the aria-selected, and aria-hidden attributes are updated on click and on keypress.

Testing

As well as testing with supported browsers and a mouse, check that:

  • You can tab to each tab, and press enter to show the content.
  • Tabbing to or past each element does not automatically activate the opening.
  • You can see (via outline or other visible change) which tab is highlighted.

Also, with common screen readers (JAWs, NVDA, VoiceOver/OSX & VoiceOver/iOS) check that:

  • You can tab and arrow to each tab.
  • You can hear which tab is which, and which is selected
    (e.g. in VoiceOver TV & Boxes, selected, tab 1 of 3.
  • You can select each tab and arrow down to the new content.
