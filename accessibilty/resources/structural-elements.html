Key structural elements

With the introduction of HTML5 and WAI-ARIA landmarks, you have to go all over the place to understand what the main layout structures should be. This page puts them all in one place. NB: this is valid as of May 2013, but worth checking the source materials (linked to below) after about September 2013!

HTML4 structural elements

These are built into HTML, already have good support from screen readers and other assistive technologies and are not made obsolete by HTML5:

Headings

Headings, levels 1-6 (<H1> - <H6>) are used to label content on the page, and screen readers use them to navigate within a page.
NB: HTML5 has defined a different way of dealing with headings, but the spec is not settled on that point and current technology still uses HTML4 style headings.
More on using headings.

Lists

Ordered (ol) and un-ordered (ul) lists are used to group things like navigation items and bulleted lists, providing a means of skipping over a list if needed.
More on using lists.

Tables

Tables should be used for tabular data, where the relationship between headings and cells matter.
More on using tables.

HTML5 structural elements

HTML5 introduces some new elements, some of which map to landmarks. However, support is not as good for those yet. Use them as per the HTML5 spec, but don't assume that assistive technology is doing anything with them yet.

Main

the main content of the body of a document or application. The main content area consists of content that is directly related to or expands upon the central topic of a document or central functionality of an application.

The start of the main should be the same place a "skip to content" link would go, and there should only be one on the page. Other elements can be nested within a main.

Article

a complete, or self-contained, composition in a document, page, application, or site and that is, in principle, independently distributable or reusable, e.g. in syndication. This could be a forum post, a magazine or newspaper article, a blog entry, a user-submitted comment, an interactive widget or gadget, or any other independent item of content.

This could be used on news pages, or other areas with fairly standard blocks of content. You would use either article or section, not both. There can be more than one per page, but please don't over do it!

Aside

a section of a page that consists of content that is tangentially related to the content around the aside element, and which could be considered separate from that content. Such sections are often represented as sidebars in printed typography.

This can be used for sidebars, but also for carousel elements at the top (it doesn't have to be to the side!). The key is that it could be considered independent from the main content.

Footer

a footer for its nearest ancestor sectioning content or sectioning root element. A footer typically contains information about its section such as who wrote it, links to related documents, copyright data, and the like.

It is likely there is a page footer (where the footer is a direct child of the body), but there can be many footers on a page if there are many article / section / aside elements (sectioning elements).

Header

introductory content for its nearest ancestor sectioning content or sectioning root element. A header typically contains a group of introductory or navigational aids.

This can apply to the whole page (which can be used with banner, below), or it can apply to a specific area of the page such as the top of a section or article.

Nav

a section of a page that links to other pages or to parts within the page: a section with navigation links.

The main and secondary navigation elements on a page should use nav, but landing pages or galleries (pages which list many links) should not mark every link-list as nav. If there is more than one on a page, you should label each (see below).

Section

a generic section of a document or application. A section, in this context, is a thematic grouping of content, typically with a heading.

Please note that a section != div! I'm not entirely sure what impact it is supposed to have on assistive technology, but at the moment Jaws will announce a section every time you navigate to one.

ARIA landmark roles

The Accessible Rich Internet Application specification (ARIA landmarks spec) has a section dedicated to landmarks, attributes that you apply to HTML elements. For example:

<form action="/search" method="post" role="search">

These provide extra hooks for assistive technology to use in navigating pages and applications. Current support is good in some screen readers (e.g. recent versions of Jaws, NVDA and VoiceOver). However, keyboard users have no main-stream way of accessing them yet.

Where ARIA-landmarks overlap with HTML5 elements, both should be used on the same element. e.g. 

<main role="main">

Banner

A region that contains mostly site-oriented content, rather than page-specific content. Typically you would apply this to the top area of each page that includes the logo & site search.

You would generally only have one banner at the top of each page.

Main

The main content of a document.

Generally this would be the target of the 'skip to content' link. You must only have one main.

Complementatary

A supporting section of the document, designed to be complementary to the main content at a similar level in the DOM hierarchy, but remains meaningful when separated from the main content.

Often used for sidebars.

Contentinfo

A large perceivable region that contains information about the parent document. Examples of information included in this region of the page are copyrights and links to privacy statements.

Basically this is for the footer.

Navigation

A collection of navigational elements (usually links) for navigating the document or related documents.

There can be more than one navigation.

Search

A landmark region that contains a collection of items and objects that, as a whole, combine to create a search facility.

Note that the search role should be applied at the form level, not just the input.

There is a complete list in the specification, but please make sure you know what impact the other ones have before using them.

Labelling landmarks

For the landmarks which can be used more than once on a page, it can help to label them.

For example, if you had this navigation element:

<nav role="navigation">
<p>Create your perfect Sky bundle:</p>
<ul>
<li><a href="fresh.html">TV & Boxes</a></li>
<li><a href="dairy.html">Broadband & Talk</a></li>
…
</ul>
</nav>

You could provide a label for that navigation:

<nav role="navigation" aria-labelledby="firstLabel">
<p>Create your perfect 
	<span id="firstLabel">Sky bundle</span>:</p>
<ul>
<li><a href="fresh.html">TV & Boxes</a></li>
<li><a href="dairy.html">Broadband & Talk</a></li>
…
</ul>
</nav>

Screen readers can then announce it as Sky bundle, navigation.
